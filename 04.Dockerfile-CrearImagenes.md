# ¿Qué es un Dockerfile?

Un **Dockerfile** es un archivo que contiene instrucciones para construir una imagen de Docker.

Estas instrucciones representan **capas (layers)** dentro de la imagen.

## Ejemplos de instrucciones en un Dockerfile

- Crear un directorio
- Exponer un puerto
- Copiar archivos
- Añadir archivos
- Ejecutar comandos de Linux
- Instalar dependencias

En resumen:

> Un Dockerfile son instrucciones de cómo construir la imagen que ejecutará nuestra aplicación.

# Objetivo de una Imagen Docker

Eventualmente tendremos nuestro código fuente:

- Funciones

- Dependencias

Lo que haremos será:

1. Tomar nuestro código fuente.
2. Empaquetarlo dentro de una imagen.
3. Ejecutarlo como un contenedor.

Ejemplo de ejecución:

```bash

```

Si queremos ejecutar una versión anterior:

```bash

```

No necesitamos instalaciones adicionales.
La imagen debe contener **todo lo necesario para ejecutarse**.

# Creación de Imágenes en Docker

## Introducción

Es momento de comenzar nuestro camino en la **creación de imágenes**.

El proceso tiene un nombre interesante: **dockerizar** o **containerizar** una aplicación.

### ¿Qué significa dockerizar?

Es el proceso de:

1. Tomar un código fuente.
2. Generar una imagen lista para ejecutarse.
3. Correrla dentro de un contenedor.

---

# Dockerfile

Para lograr esto necesitamos crear un archivo llamado:

```
Dockerfile
```

Un **Dockerfile** es:

- Un archivo de texto
- Con instrucciones
- Que Docker usa para construir una imagen
- Funciona como un *blueprint* o plano de construcción

Cada instrucción que coloquemos generará una **capa (layer)**.

---

# Layers (Capas)

Cada línea del Dockerfile crea una capa.

Cuando descargamos una imagen, lo que realmente se descargan son **layers**.

Ventajas:

- Se pueden reutilizar entre imágenes
- Mejoran el rendimiento
- Reducen descargas innecesarias

Ejemplos de instrucciones que generan layers:

- Configuración de variables de entorno
- Ejecución de comandos bash
- Copiar archivos
- Definir un volumen
- Exponer puertos
- Definir un `ENTRYPOINT`
- Definir un `CMD`

# Implementando Cron

En `app.js`, usar algo similar a esto:

```javascript
const { CronJob } = require('cron');

let times = 0;

const job = new CronJob('* * * * * *', () => {
  console.log('Tarea ejecutándose cada segundo');
});

job.start();
```

Cada asterisco representa:

```
* * * * * *
| | | | | |
| | | | | Día de la semana
| | | | Mes
| | | Día del mes
| | Hora
| Minuto
Segundo
```

---

# Ejecutar cada 5 segundos

Para que se ejecute en el segundo 5, 10, 15, 20, etc:

```javascript
const { CronJob } = require('cron');

let times = 0;

const job = new CronJob('*/5 * * * * *', () => {
  times++;
  console.log(`Tick cada 5 segundos: ${times}`);
});

job.start();
```

Ejecutar:

```bash
npm start
```

Para detener:

```bash
Ctrl + C
```

# ¿Por Qué Crear una Imagen Docker?

Imaginemos que nuestra aplicación:

- Consume una API
- Lee datos de una base de datos
- Inserta información en otra base
- Ejecuta procesos de sincronización

En un entorno real necesitamos:

- Desplegarla en un servidor
- Mantener versiones
- Actualizar fácilmente
- No depender de instalaciones manuales

La solución ideal es crear una **imagen Docker**.

Luego simplemente ejecutamos:

```bash
docker run nombre-imagen
```

Sin preocuparnos por:

- Instalar Node
- Instalar dependencias
- Configurar el sistema operativo

---

# Dependencias de Nuestra Aplicación

Nuestra aplicación necesita:

- Node
- El paquete `cron`
- Sus dependencias internas
- Un sistema operativo donde ejecutarse

Si en el servidor no está instalado Node, este comando fallará:

```bash
node app.js
```

Por eso necesitamos una imagen que contenga:

- Linux
- Node
- Nuestra aplicación
- Sus dependencias

---

# Todo Comienza con el Dockerfile

El archivo clave es:

```
Dockerfile
```

Cada línea dentro del Dockerfile es una instrucción.

Cada instrucción crea un **layer (capa)**.

Cuando descargamos una imagen, Docker descarga múltiples capas en paralelo.

Si una capa no cambia, Docker la reutiliza desde caché.

---

# Ejemplo Real: Imagen Oficial de Node

En Docker Hub podemos ver la imagen oficial de Node.

Tiene:

- Miles de millones de descargas
- Múltiples tags (versiones)
- Diferentes variantes (Debian, Alpine, etc.)

Ejemplo de tag:

```
node:19.2-alpine3.16
```

## ¿Qué significa?

- `19.2` → versión de Node
- `alpine3.16` → versión de Alpine Linux
- Alpine → distribución Linux muy ligera

Las versiones Alpine son más pequeñas que otras variantes.

---

# Paso 1: FROM (Imagen Base)

Toda imagen comienza con:

```dockerfile
FROM node:19.2-alpine3.16
```

Esto significa:

- Partimos de una imagen existente
- Contiene Alpine Linux
- Tiene Node preinstalado

En el 99% de los casos trabajaremos sobre una imagen base ya existente.

---

# Paso 2: WORKDIR

Ahora definimos el directorio de trabajo:

```dockerfile
WORKDIR /app
```

Esto es equivalente a hacer:

```bash
cd /app
```

Todos los paths relativos ahora estarán dentro de `/app`.

La imagen base ya incluye varias carpetas como:

- `/usr`
- `/lib`
- `/app`
- entre otras

---

# Estado Actual de la Imagen

En este momento tenemos:

- Linux (Alpine)
- Node instalado
- Directorio `/app` seleccionado
- Pero NO tenemos nuestra aplicación todavía

Necesitamos copiar nuestros archivos.

---

# Paso 3: COPY

Debemos copiar:

- `app.js`
- `package.json`

Instrucción:

```dockerfile
COPY app.js package.json ./
```

Explicación:

- `app.js` → archivo fuente
- `package.json` → dependencias y script start
- `./` → destino (relativo a `/app`)

Como ya usamos `WORKDIR /app`, el destino real será:

```
/app
```

Si no hubiéramos usado `WORKDIR`, tendríamos que escribir:

```dockerfile
COPY app.js package.json /app
```

---

# Estado Actual del Dockerfile

Hasta ahora tenemos:

```dockerfile
FROM node:19.2-alpine3.16

WORKDIR /app

COPY app.js package.json ./
```

Con esto ya:

- Partimos de Node + Alpine
- Nos movimos a `/app`
- Copiamos nuestra aplicación

---

- Instalar dependencias

# Construcción de la Imagen – Docker Build y Ejecución

## Estado Actual del Dockerfile

En la clase anterior nos quedamos con este Dockerfile:

```dockerfile
FROM node:19.2-alpine3.16

WORKDIR /app

COPY app.js package.json ./
```

### ¿Qué hemos hecho hasta ahora?

1. **FROM** → Estamos tomando una versión de Linux (Alpine) que ya trae Node preinstalado.
2. **WORKDIR /app** → Nos movimos al directorio `/app`.
3. **COPY app.js package.json ./** → Copiamos:
   - `app.js`
   - `package.json`

   Desde nuestro proyecto local hacia `/app` dentro de la imagen.

---

## Estado Interno de la Imagen en Este Punto

En este momento, dentro de la imagen solo existen:

- `app.js`
- `package.json`

No existen todavía:

- `node_modules`
- Dependencias instaladas
- `package-lock.json` reconstruido

Es decir, la aplicación todavía **no puede ejecutarse**, porque faltan las dependencias.

---

# Instalación de Dependencias (npm install)

En una aplicación Node, las dependencias se instalan con:

```bash
npm install
```

Este comando hace lo siguiente:

1. Lee `package.json`
2. Revisa:
   - `dependencies`
   - `devDependencies`
3. Instala las dependencias
4. Crea o reconstruye:
   - `package-lock.json`
   - Carpeta `node_modules`

Esto mismo debemos hacerlo dentro de nuestra imagen.

---

# Paso 4: RUN npm install

Para ejecutar comandos dentro del Dockerfile usamos:

```dockerfile
RUN comando
```

Entonces agregamos:

```dockerfile
RUN npm install
```

Esto es equivalente a entrar al contenedor y ejecutar manualmente:

```bash
npm install
```

Si quisiéramos ejecutar múltiples comandos:

```dockerfile
RUN npm install && echo "Instalación completada"
```

Las instrucciones del Dockerfile se ejecutan de manera secuencial:

1. FROM
2. WORKDIR
3. COPY
4. RUN
5. ...

Cada una crea un layer.

---

# Paso 5: CMD (Comando de Inicio)

Ya tenemos:

- Node
- Nuestra aplicación
- Dependencias instaladas

Ahora necesitamos decirle a Docker:

> ¿Qué debe ejecutar cuando se inicie el contenedor?

Para eso usamos:

```dockerfile
CMD ["node", "app.js"]
```

Esto indica que cuando hagamos:

```bash
docker run nombre-imagen
```

Se ejecutará automáticamente:

```bash
node app.js
```

También podríamos usar:

```dockerfile
CMD ["npm", "start"]
```

Ambos funcionarían porque ya tenemos el script definido en `package.json`.

---

# Dockerfile Completo

```dockerfile
FROM node:19.2-alpine3.16

WORKDIR /app

COPY app.js package.json ./

RUN npm install

CMD ["node", "app.js"]
```

Si ejecutas nuevamente:

¿Por qué?

Porque Docker:

- Revisa si hubo cambios
- Si no hay cambios → reutiliza layers desde caché

### Buena práctica

Colocar arriba los comandos que cambian menos.

Porque si modificas una instrucción intermedia:

- Todas las capas siguientes se reconstruyen.

# Reconstrucción de Imágenes y Manejo de Tags en Docker

## ¿Por Qué Reconstruir una Imagen?

Existen varias razones por las que necesitaremos reconstruir una imagen:

1. ❌ Nos equivocamos en algún paso del Dockerfile.
2. 🔄 Hicimos cambios en el código.
3. 🛠 Hicimos cambios en el Dockerfile.
4. 🚀 Necesitamos actualizar comportamiento (por ejemplo: cambiar lógica de ejecución cada 5 segundos).

Estos son los casos más comunes en la vida real.

---

# Cómo Funciona el Cache de Docker

Supongamos que hacemos un cambio en el código.

Guardamos los cambios y ejecutamos nuevamente:

```bash
docker build -t cron-ticker .
```

Docker empieza a reconstruir.

Durante el proceso verás algo como:

```
Step 1/5
Step 2/5
Step 3/5
...
```

### ¿Qué ocurre internamente?

Docker revisa cada instrucción del Dockerfile y compara:

- Si esa instrucción cambió
- Si los archivos que esa instrucción usa cambiaron

Si no hubo cambios → usa el **cache**
Si hubo cambios → reconstruye ese layer y todos los siguientes

---

## Ejemplo: Cambio en el Código

Si solo modificas `app.js`:

- FROM → cache
- WORKDIR → cache
- COPY package.json → cache
- RUN npm install → cache (si package.json no cambió)
- COPY app.js → se reconstruye
- Capas siguientes → se reconstruyen

🔎 Regla importante:

> Si un layer cambia, todos los layers posteriores también se reconstruyen.

---

# Optimización del Orden de Instrucciones

Una buena práctica es colocar arriba las instrucciones que cambian menos.

Por ejemplo:

```dockerfile
FROM node:19.2-alpine3.16

WORKDIR /app

COPY package.json ./

RUN npm install

COPY app.js ./

CMD ["node", "app.js"]
```

### ¿Por qué así?

Porque:

- `package.json` cambia poco
- `app.js` cambia más seguido

Entonces:

- Si solo cambia `app.js`
- No se vuelve a ejecutar `npm install`
- La construcción es mucho más rápida

Esto es una práctica profesional real.

---

# Listar Imágenes

Para ver las imágenes:

```bash
docker image ls
```

Aquí ocurre algo interesante.

Cada vez que reconstruyes sin cambiar el tag, Docker:

- Mantiene la imagen anterior
- Crea una nueva imagen
- Reasigna el `latest`

Las imágenes anteriores quedan sin nombre (dangling images).

Esto puede generar acumulación innecesaria.

---

# Entendiendo el Tag "latest"

Si construyes así:

```bash
docker build -t cron-ticker .
```

Docker asigna automáticamente:

```
cron-ticker:latest
```

Cada nueva construcción reemplaza el `latest`, pero las anteriores siguen existiendo con su ID.

---

# Versionado Correcto

En lugar de depender solo de `latest`, debemos versionar.

Ejemplo:

```bash
docker build -t cron-ticker:1.0.0 .
```

Si ejecutamos nuevamente el mismo comando sin cambios:

- Todo estará en cache
- No reconstruirá innecesariamente

Ahora veremos:

```bash
docker image ls
```

Y tendremos:

```
cron-ticker   latest
cron-ticker   1.0.0
```

Ambos apuntan a la misma imagen (mismo IMAGE ID).

---

# Crear Nuevos Tags Sin Reconstruir

No es necesario reconstruir para crear otro nombre.

Podemos usar:

```bash
docker image tag cron-ticker:1.0.0 cron-ticker:buffalo
```

Ahora:

```bash
docker image ls
```

Veremos:

```
cron-ticker   1.0.0
cron-ticker   buffalo
cron-ticker   latest
```

Todos apuntando al mismo IMAGE ID.

Esto es solo un alias.

---


# Planes en Docker Hub

Docker Hub ofrece:

### Plan gratuito
- Repositorios públicos ilimitados
- 1 repositorio privado
- Límite de pulls por período
- Tokens de acceso personales

### Plan pago (~5 USD/mes)
- Repositorios privados ilimitados
- Mayor cantidad de pulls
- Más funcionalidades empresariales

También existen alternativas:

- Registro privado en DigitalOcean
- Azure Container Registry
- GitHub Container Registry
- Entre otros

Pero aquí usaremos Docker Hub.

---

# Paso 1: Crear un Repositorio en Docker Hub

1. Iniciar sesión en Docker Hub.
2. Ir a **Repositories**.
3. Crear nuevo repositorio.

Nombre sugerido:

```
cron-ticker
```

Puede ser:

- Público
- Privado

Una vez creado, Docker Hub mostrará algo como:

```
docker push usuario/cron-ticker
```

Ese será el nombre completo que debemos usar.

---

# Concepto Importante: Nombre Completo de Imagen

Una imagen en Docker Hub tiene el formato:

```
usuario/nombre-imagen:tag
```

Ejemplo:

```
clarito/cron-ticker:latest
```

Ese prefijo (`usuario/`) es obligatorio para poder subirla.

---

# Paso 2: Renombrar la Imagen Local

Primero debemos crear un tag con el nombre del repositorio remoto.

Usamos:

```bash
docker image tag cron-ticker:latest usuario/cron-ticker:latest
```

Si no especificamos `:latest`, Docker lo asume por defecto.

Verificamos:

```bash
docker image ls
```

Ahora veremos algo como:

```
cron-ticker             latest
usuario/cron-ticker     latest
```

Ambas apuntan al mismo IMAGE ID.

Solo cambiamos el nombre.

---

# Paso 3: Intentar Hacer Push (Sin Login)

Si intentamos:

```bash
docker push usuario/cron-ticker:latest
```

Recibiremos un error similar a:

```
access denied
```

¿Por qué?

Porque no estamos autenticados.

---

# Paso 4: Autenticación

Iniciamos sesión:

```bash
docker login
```

Docker pedirá:

- Username
- Password

Por defecto intenta autenticarse contra:

```
https://index.docker.io/v1/
```

Una vez autenticado correctamente:

```
Login Succeeded
```

También es recomendable usar **Access Tokens** en vez de contraseña directa para mayor seguridad.

---

# Paso 5: Hacer Push

Ahora sí:

```bash
docker push usuario/cron-ticker:latest
```

Docker empezará a subir:

- Cada layer
- Mostrará progreso
- Subirá solo layers que no existan ya en el registro

Si todo termina correctamente:

```
latest: digest: sha256:...
```

---

# Verificar en Docker Hub

Recargamos el navegador.

Ahora veremos:

- Nuestra imagen
- Tag `latest`
- Tamaño comprimido (ejemplo: ~53MB)
- Arquitectura (amd64, arm64, etc.)

---

# Nota Sobre Arquitectura

Si estás en:

- Mac M1 → ARM64
- PC tradicional → AMD64

Docker indicará la arquitectura de la imagen.

Esto es importante cuando trabajemos con imágenes multi-arquitectura más adelante.

# Seguridad Automática (Escaneo)

Docker Hub permite:

- Escaneo automático de vulnerabilidades
- Integración con herramientas como Snyk
- Verificación de dependencias inseguras

Este proceso puede tardar unos minutos.

Es una funcionalidad muy útil.

---

# Importante: Problema con latest

Cada vez que hacemos push a `latest`:

- Se pierde la referencia anterior
- Solo queda la última versión como latest

Esto es similar a lo que pasó localmente con imágenes sin tag.

Por eso es buena práctica subir versiones específicas.

---

# Subir Otro Tag (Ejemplo: castor)

Primero creamos el tag remoto:

```bash
docker image tag cron-ticker:castor usuario/cron-ticker:castor
```

Luego hacemos push:

```bash
docker push usuario/cron-ticker:castor
```

Notarás algo interesante:

Docker dirá que muchos layers:

```
Layer already exists
```

¿Por qué?

Porque los layers son idénticos a los que ya subimos.

Docker solo crea la nueva referencia del tag.

Es extremadamente eficiente.

---

# Verificar Nuevamente en Docker Hub

Ahora veremos:

- latest
- castor
- otros tags que subamos

Todos apuntando posiblemente al mismo digest si no cambió la imagen.

---

# Concepto Clave

Docker no sube imágenes completas cada vez.

Sube layers.

Si un layer ya existe en el registro:

- No lo vuelve a subir.
- Solo referencia el existente.

Esto hace que los pushes sean rápidos después del primero.

# Ejecutar la Imagen Directamente Desde Docker Hub

En esta clase vamos a hacer algo muy importante:

> Ejecutar la imagen que subimos a Docker Hub directamente desde la nube, sin tenerla localmente.

Esto simula estar en una máquina completamente nueva.

---

# Paso 1: Verificar Imágenes Locales

Primero listamos nuestras imágenes:

```bash
docker image ls
```

Aquí veremos:

- cron-ticker:latest
- cron-ticker:castor
- cron-ticker:1.0.0
- etc.

Si ejecutáramos la imagen en este momento:

```bash
docker container run usuario/cron-ticker:castor
```

Docker usaría la imagen local.

No iría a la nube.
No descargaría nada.
No estaríamos probando realmente el registro remoto.

---

# Paso 2: Limpiar Imágenes Locales

## Eliminar imágenes sin nombre (dangling)

```bash
docker image prune
```

Esto elimina:

- Imágenes sin tag
- Imágenes intermedias
- Capas huérfanas

Después:

```bash
docker image ls
```

Las imágenes con nombre aún permanecerán.

---

## Eliminar TODAS las imágenes

Si queremos simular una máquina limpia completamente:

```bash
docker image prune -a
```

⚠ Esto elimina todas las imágenes no utilizadas por contenedores.

Confirmamos con `y`.

Ahora:

```bash
docker image ls
```

No debería aparecer ninguna imagen.

Perfecto. Estamos en un entorno limpio.

---

# Paso 3: Ejecutar Imagen Desde Docker Hub

Ahora ejecutamos directamente desde el registro remoto:

```bash
docker container run usuario/cron-ticker:castor
```

¿Qué ocurre ahora?

1. Docker busca la imagen localmente.
2. No la encuentra.
3. Consulta Docker Hub.
4. Encuentra el tag.
5. Descarga los layers.
6. Construye la imagen localmente.
7. Ejecuta el contenedor.

Verás algo como:

```
Unable to find image locally
Pulling from usuario/cron-ticker
...
Downloading layers...
```

Luego comienza a ejecutarse la aplicación:

```
Tick cada múltiplo de cinco
```

🎉 Funciona aunque no tengamos nada local.

---

# Detener el Contenedor

Normalmente:

```bash
Ctrl + C
```

Si se queda colgado:

```bash
docker container ls
```

Luego:

```bash
docker container rm -f ID_DEL_CONTENEDOR
```

# Concepto Clave

Todo este proceso que hemos hecho:

- FROM
- WORKDIR
- COPY
- RUN
- CMD

Es independiente del tamaño de la aplicación.

Da igual si es:

- Un microservicio pequeño
- Una API grande
- Una aplicación en Java
- Una aplicación en Python
- Un sistema complejo

La mecánica es la misma.

Lo único que cambia son los pasos específicos de instalación:

Ejemplo Python:

```dockerfile
RUN pip3 install -r requirements.txt
```

Ejemplo Java:

```dockerfile
RUN mvn package
```

Los pasos de abajo dependen de la tecnología.
La estructura general es siempre igual.
