# 🐳 Sección 4 — Aplicaciones con Múltiples Contenedores (Docker Compose)

## 🎯 Objetivo de la sección

En esta sección aprenderemos a trabajar con **aplicaciones compuestas por múltiples contenedores** utilizando una herramienta llamada:
```
Docker Compose
```

---

## 📦 ¿Qué es Docker Compose?

Docker Compose permite definir y ejecutar múltiples contenedores usando **un solo archivo de configuración**.

En lugar de ejecutar muchos comandos manualmente, utilizamos un archivo llamado:

```
docker-compose.yml
```

Este archivo describe toda la infraestructura necesaria para una aplicación.

---

## 🚨 Problema sin Docker Compose

Supongamos que tenemos:

- 🗄️ Un contenedor de base de datos
- 🔍 Un contenedor para explorar la base de datos
- 🔁 Una réplica de la base de datos
- 🚀 Un contenedor con nuestra aplicación

Sin Compose tendríamos que ejecutar manualmente:

- Un comando para cada contenedor
- Configurar redes
- Crear volúmenes
- Definir conexiones
- Recordar múltiples configuraciones

Ejemplo del problema:
```
docker run ...
docker run ...
docker run ...
docker network create ...
docker volume create ...
```

❌ Fácil olvidar pasos
❌ Difícil mantener
❌ Poco escalable

---

## ✅ Solución: Docker Compose

Docker Compose permite definir todo en un solo archivo.

Dentro del archivo podemos especificar:

- Versión
- Servicios (contenedores)
- Redes
- Volúmenes
- Variables de entorno
- Dependencias entre servicios

Cada **servicio** representa un contenedor independiente.

---

## 🧱 Ejemplo conceptual

Arquitectura:
```
Aplicación
│
├── Database Container
├── Database Replica
├── DB Explorer
└── Backend Service
```

Docker Compose se encarga automáticamente de:

✅ Crear contenedores
✅ Crear redes
✅ Crear volúmenes
✅ Conectar servicios
✅ Descargar imágenes necesarias

Todo con **un solo comando**.

---

## ⚡ Ventaja principal

En lugar de múltiples pasos manuales:
```
docker run ...
docker run ...
docker run ...
```

Solo ejecutamos:
```
docker compose up
```

Y todo el entorno queda listo.

---

## 👨‍💻 Caso real de uso

Imagina que llega un nuevo desarrollador al equipo.

Sin Docker Compose tendría que:

- Instalar base de datos
- Configurar servicios
- Crear redes
- Importar datos
- Configurar conexiones

Proceso lento y propenso a errores.

---

### Con Docker Compose

Solo necesita:

1. Clonar el repositorio
2. Ejecutar:
```
docker compose up
```

✅ Entorno completo listo
✅ Versiones correctas
✅ Configuración idéntica para todo el equipo

---

## 💾 Manejo de Datos

Compose también permite:

- Crear volúmenes automáticamente
- Usar volúmenes externos
- Cargar semillas de base de datos
- Restaurar backups

Esto facilita trabajar con datos preconfigurados.

# 🧪 Laboratorio — PostgreSQL + PgAdmin con Docker

## 📦 Imágenes a utilizar

- Imagen oficial de PostgreSQL

PgAdmin es un gestor web muy popular para administrar bases de datos PostgreSQL.

---

## 🔎 Puertos expuestos en PgAdmin

La imagen expone:

```
80
443
```

- `80` → acceso web HTTP
- `443` → HTTPS

El puerto **80** será el utilizado para acceder desde el navegador.

---

## 🧱 Paso 1 — Crear volumen para PostgreSQL

Crear un volumen para persistir la información:
```
docker volume create postgres-db
```

Esto evita perder datos cuando el contenedor se elimine.

---

## 🐘 Paso 2 — Crear contenedor PostgreSQL

Ejecutar PostgreSQL:
```
docker container run
--name postgres-db
-e POSTGRES_PASSWORD=123456
-v postgres-db:/var/lib/postgresql/data
-d postgres
```

⚠️ Importante:

No se publica ningún puerto.

Esto significa que:

❌ No podremos conectarnos externamente
✅ Solo otros contenedores podrán acceder

---

## 🌐 Paso 3 — Ejecutar PgAdmin

Crear contenedor PgAdmin:
```
docker container run
--name pgadmin
-e PGADMIN_DEFAULT_EMAIL=superman@google.com

-e PGADMIN_DEFAULT_PASSWORD=123456
-p 8080:80
-d dpage/pgadmin4
```

---

## 🌍 Paso 4 — Acceder a PgAdmin

Abrir navegador:
```
http://localhost:8080
```

Credenciales:

- Email: `superman@google.com`
- Password: `123456`

---

Dentro de PgAdmin:

1. Click derecho → **Servers**
2. Seleccionar **Register → Server**

---

### 🔗 Connection

| Port | 5432 |
| Database | postgres |

Crear carpeta:
```

```

Abrir la carpeta en **Visual Studio Code**.

---

Crear archivo:
```
docker-compose.yml
```

⚠️ También puede llamarse:
```
docker-compose.yaml
```

Ambos funcionan igual.

---

## 🧠 ¿Qué es Docker Compose?

Docker Compose es una herramienta que permite:

- Definir múltiples contenedores
- Compartir configuraciones
- Crear redes automáticamente
- Manejar volúmenes
- Levantar toda la aplicación con un comando

## 🔢 Paso 4 — Definir versión

Siempre iniciamos indicando la versión:

## 🧱 Paso 5 — Definir servicios

Todos los contenedores viven dentro de:
```

```

# 🐘 Servicio PostgreSQL

```

```

El nombre `database` será:

✅ Nombre interno del servicio
✅ Nombre DNS dentro de la red Docker

```

```

Buena práctica para identificar contenedores fácilmente.

---

## 🖼️ Imagen

```
image: postgres:15.1
```

---

## 💾 Volumen de persistencia

```

    postgres-db:/var/lib/postgresql/data
```

Esto equivale al comando manual:
```
docker run -v postgres-db:/var/lib/postgresql/data
```

---

## 🌎 Variables de entorno
```

    POSTGRES_PASSWORD: 123456
```

Equivale a:
```

```

# 🐳 Docker Compose — Uso de Volúmenes Externos

## 🎯 Objetivo de la clase

Aprender a:

- Detener correctamente servicios de Docker Compose
- Usar un **volumen externo existente**
- Entender cuándo es necesario recrear contenedores
- Recuperar datos persistentes

---

## 🛑 Paso 1 — Detener contenedores en ejecución

Cuando ejecutamos:
```
docker compose up
```

Los contenedores quedan **adjuntos a la terminal** (*attached mode*).

Para detenerlos:

```
CTRL + C
```

Esto detiene todo el grupo de contenedores.

---

## ⚠️ Concepto importante

Detener ≠ eliminar

Aunque el contenedor se detenga:

✅ Volúmenes siguen existiendo
✅ Datos permanecen guardados

---

## 🔎 Verificar volúmenes existentes
```
docker volume ls
```

Ejemplo:
```
postgres-db
postgresql-pgadmin_postgres-db
```

Docker Compose creó automáticamente un volumen nuevo.

## 🎯 Problema detectado

## 🌎 Variables de entorno oficiales de Mongo

Mongo permite crear automáticamente un usuario root usando:
```
MONGO_INITDB_ROOT_USERNAME

```

⚠️ Estos nombres deben respetarse exactamente.

---

## 🧱 Configuración en docker-compose.yml

```yaml
services:
  db:
    container_name: pokemon-db
    image: mongo:6.0

    command: ["--auth"]

    environment:
      MONGO_INITDB_ROOT_USERNAME: fernando
      MONGO_INITDB_ROOT_PASSWORD: 123456

    volumes:
      - pokeball:/data/db

    ports:
      - "27017:27017"

    restart: always

volumes:
  pokeball:
    external: false
```
## ⚠️ Problema común (MUY IMPORTANTE)

Después de agregar autenticación:

La conexión falla ❌

Aunque usuario y contraseña sean correctos.

Mongo ejecuta:

```

MONGO_INITDB_ROOT_PASSWORD
```

👉 solo la primera vez que se inicializa la base de datos.

Pero nuestra base ya existe dentro del volumen.

```
docker compose down
```

```

```
⚠️ Esto elimina todos los datos.
4️⃣ Levantar nuevamente
```
docker compose up -d
```
Ahora Mongo se inicializa desde cero ✅

Mongo Express necesita que MongoDB esté activo.

Docker Compose crea automáticamente:

✅ Una red

o el nombre del contenedor.

```

```
NO usamos:
```

```
porque los contenedores viven en otra red.

# 🧹 Limpieza del Entorno Docker (Post Multi-Container App)

## 🎯 Objetivo de la clase

En el material adjunto se proporciona:

- ✅ Archivo `.env`
- ✅ Archivo `docker-compose.yml`

Con estos archivos basta con:

```bash
docker compose up
```
para levantar toda la aplicación.
## 📁 Cómo ejecutar el proyecto

1. Crear un directorio nuevo.
2. Colocar dentro:
```
.env
docker-compose.yml
```
3. Ejecutar
```
docker compose up
```
Y toda la aplicación se levanta automáticamente.
## ⚠️ ¿Por qué hacer esta clase?

Porque aunque ejecutemos:
```
docker compose down
```
Docker NO elimina todo.

Todavía quedan recursos ocupando espacio:
* Volúmenes
* Imágenes
* Redes
* Builders
Por eso necesitamos hacer limpieza manual.
## 🧱 Paso 1 — Bajar contenedores
```
docker compose down
```
Esto elimina:

✅ Contenedores
✅ Red del proyecto

Pero **NO elimina volúmenes.**
## 💾 Paso 2 — Revisar volúmenes

```
docker volume ls
```
Ejemplo:
```
pokemonapp_pokeball
```
Aunque el contenedor se eliminó,
la base de datos sigue existiendo dentro del volumen.

## 🔎 Importante

Si vuelves a ejecutar:
```
docker compose up
```
La información seguirá ahí.

Porque los datos viven en el volumen.

## 🗑️ Paso 3 — Eliminar volumen específico

Forma recomendada:
```
docker volume rm NOMBRE_DEL_VOLUMEN
```
Ejemplo:
```
docker volume rm pokemonapp_pokeball
```
## ⚠️ Eliminación completa (cuidado)
```
docker volume prune
```
Esto elimina todos los volúmenes no utilizados.

⚠️ Se pierde toda la data almacenada.

## 🌐 Paso 4 — Revisar redes

```
docker network ls
```
Normalmente Docker Compose elimina su red automáticamente.

## 📦 Paso 5 — Revisar imágenes
```
docker image ls
```
Ejemplo:

* mongo
* mongo-express
* pokemon-app

Estas imágenes pueden ocupar:
```
Mongo        ≈ 600MB
App          ≈ 400MB
Otros        ≈ cientos de MB
```
## 🗑️ Eliminar imágenes

Por ID:
```
docker image rm IMAGE_ID
```
O varias:
```
docker image rm id1 id2 id3
```
Verificación:
```
docker image ls
```
## 🏗️ Paso 6 — Revisar Builders
Docker también guarda builders internos.

Consultar:
```
docker builder ls
```
Aquí aparecen:

* builders por arquitectura
* configuraciones de build

Esto será importante cuando creemos imágenes multi-arquitectura.

## 🧠 Idea importante

Docker acumula con el tiempo:

* imágenes
* capas
* volúmenes
* builds

Si no se limpian → consume mucho espacio.

## ✅ Resultado Final

Después de la limpieza:
```
atu✔ Sin contenedorese
✔ Sin volúmenes
✔ Sin redes
✔ Sin imágenes
✔ Entorno limpio
```
