# 🐳 ¿Qué es Docker y por qué debería aprenderlo?

Antes de comenzar con comandos y práctica, es importante entender **el problema real que Docker viene a solucionar**.

Esta es una explicación conceptual clave para comprender por qué Docker se volvió una herramienta esencial en la industria del software.

---

# 😵 El Problema Antes de Docker

Imagina que un nuevo desarrollador entra a tu proyecto.

Lo primero que le dices es:

> “Por favor instala estas herramientas para poder trabajar.”

Por ejemplo:

- MariaDB
- Redis
- MongoDB
- Node.js
- Otras dependencias

El desarrollador instala todo… pero aparece el primer problema.

---

## ⚠️ Problema 1 — Versiones específicas

Luego descubres que necesitas versiones exactas:

- MariaDB **10.3**
- Redis versión antigua
- MongoDB **4.x**
- Dependencias legacy

¿Por qué?

- La aplicación es antigua (*legacy*)
- Existen incompatibilidades
- Migraciones pendientes
- Código dependiente de versiones específicas

Resultado:

👉 La instalación ya no es simple.

---

## 💻 Problema 2 — Diferentes Sistemas Operativos

Ahora el desarrollador tiene:

- 🖥️ Una computadora Linux en la oficina
- 💻 Una laptop Windows en casa
- 🍎 Tal vez una Mac

Empiezan los errores clásicos:

- Funciona en Linux pero no en Windows
- No existe la misma versión en Mac
- La distribución Linux no soporta el paquete requerido

Aparece el famoso problema:

> ✅ *“En mi máquina funciona”*
> ❌ *“En producción no funciona”*

---

## 🔥 Problema 3 — Múltiples Aplicaciones

Ahora el desarrollador debe mantener:

- Aplicación A → versiones antiguas
- Aplicación B → versiones nuevas

Ejemplo:

| Aplicación | Redis | MongoDB |
|---|---|---|
| App 1 | v5 | v4 |
| App 2 | v7 | v5 |

Intentar mantener todo esto en una misma computadora se vuelve un caos.

---

# 🖥️ ¿Y las Máquinas Virtuales?

Una posible solución histórica fueron las **máquinas virtuales (VM)**.

Pero tienen desventajas importantes:

- Consumen muchos recursos
- Ejecutan un sistema operativo completo
- Son pesadas
- Inician lentamente
- Difíciles de mover o compartir

Una VM incluye cosas innecesarias como:

- Interfaces gráficas
- Servicios del sistema
- Aplicaciones preinstaladas

👉 Solo queremos ejecutar una aplicación, no otro sistema operativo entero.

---

# 🚀 La Revolución: Docker

Docker introduce un concepto diferente.

## 📦 Imágenes Docker

Una **imagen** puede entenderse como:

> 📸 Una fotografía lista para ejecutar una tecnología específica.

Ejemplo:

- Imagen de MariaDB 10.3
- Imagen de Redis 5
- Imagen de MongoDB 4

Estas imágenes contienen **todo lo necesario para funcionar**.

---

## 🧱 Contenedores

Cuando ejecutamos una imagen, Docker crea un:

👉 **Contenedor**

Un contenedor es:

- Aislado
- Independiente
- Reproducible
- Ligero

Cada contenedor tiene su propio entorno.

---

## ✅ Ventajas Clave

Con Docker puedes:

- Ejecutar múltiples versiones simultáneamente
- Aislar aplicaciones entre sí
- Evitar conflictos de dependencias
- Ejecutar todo con un solo comando
- Eliminar sin dejar basura en el sistema

Ejemplo:

Puedes correr al mismo tiempo:

- 3 bases de datos Mongo diferentes
- Varias versiones de Redis
- Varias aplicaciones

Todo en la misma computadora.

---

## 🌍 Independiente del Sistema Operativo

Docker garantiza que:

> La aplicación funcionará igual en cualquier entorno.

- Linux ✅
- Windows ✅
- macOS ✅
- Servidores remotos ✅

Docker abstrae las diferencias del sistema operativo.

---

# ⚡ Contenedores vs Máquinas Virtuales

| Característica | Máquina Virtual | Docker |
|---|---|---|
| Sistema operativo completo | ✅ | ❌ |
| Consumo de recursos | Alto | Bajo |
| Tiempo de arranque | Minutos | Segundos |
| Portabilidad | Limitada | Alta |
| Rendimiento | Menor | Mayor |

# 🐳 Primeros Pasos Prácticos — `docker pull` y `docker run`

Ahora sí comenzamos oficialmente con la **parte práctica de Docker**.

---

# 📥 Comando `docker pull`

El comando:

    docker pull

sirve para:

👉 **Descargar una imagen desde un repositorio remoto hacia tu computadora**.

En otras palabras:

> Traer una imagen lista para usarse.

---

## 📦 ¿Qué es una Imagen Docker?

Una **imagen** es:

> Un archivo construido por capas que contiene todo lo necesario para ejecutar una aplicación.

Incluye:

- Dependencias
- Configuraciones
- Archivos binarios
- Scripts de ejecución
- Librerías necesarias
- Entorno de ejecución

✅ Todo listo para funcionar.

Puedes imaginar una imagen como:

📸 **Una fotografía completa de un entorno funcionando**.

---

# ▶️ Descargar tu Primera Imagen

Abre tu terminal (CMD, PowerShell, Terminal Linux o Mac) y ejecuta:

    docker pull hello-world

---

## 🔎 ¿Qué sucede internamente?

Docker realiza varios pasos:

1. Busca la imagen localmente
2. Si no existe → va a Internet
3. Descarga la imagen
4. La guarda en tu máquina

Verás algo similar a:

    Using default tag: latest

---

## 🏷️ ¿Qué significa `latest`?

`latest` es una **etiqueta (tag)**.

⚠️ Importante:

`latest` **NO siempre significa la versión más nueva**.

Simplemente es una etiqueta llamada *latest*.

En imágenes oficiales suele apuntar a la versión más reciente, pero en imágenes personalizadas puede significar cualquier cosa.

Más adelante veremos tags en profundidad.

---

# ▶️ Ejecutar una Imagen

Una vez descargada la imagen, podemos ejecutarla:

    docker container run hello-world

---

## 🧠 ¿Qué acaba de pasar?

Docker hizo automáticamente:

1. Crear un contenedor
2. Montar la imagen dentro del contenedor
3. Ejecutarlo
4. Mostrar el resultado
5. Finalizar el contenedor

Resultado esperado:

    Hello from Docker!

Esto confirma que Docker funciona correctamente ✅

---

# 📦 ¿Qué es un Contenedor?

Un **contenedor** es:

> Una instancia de una imagen ejecutándose en un entorno aislado.

En términos simples: Imagen → se ejecuta → Contenedor


- Imagen = plantilla
- Contenedor = instancia en ejecución

---

# ⚙️ Docker Engine

Cuando ejecutas:

    docker container run hello-world

Realmente estás diciendo:

> Docker Engine, crea y ejecuta un contenedor usando esta imagen.

Docker Engine es el servicio en segundo plano encargado de:

- Crear contenedores
- Ejecutarlos
- Administrarlos
- Distribuir imágenes

---

# 🖥️ Observándolo en Docker Desktop

Si abres **Docker Desktop** verás:

- Un nuevo contenedor creado
- Nombre generado automáticamente
- Estado: `Exited`

Docker asigna nombres aleatorios (científicos o hackers famosos).

---

## ❓ ¿Por qué aparece como `Exited`?

Porque el contenedor:

1. Se inicia
2. Ejecuta su tarea
3. Termina
4. Se detiene automáticamente

El contenedor cumplió su trabajo.

---

# 🔁 Ejecutar Nuevamente

Si ejecutas otra vez:

    docker container run hello-world

Docker:

✅ NO vuelve a descargar la imagen
✅ Solo crea otro contenedor nuevo

---

## Probar nuevamente `pull`

Ejecuta otra vez:

    docker pull hello-world

Ahora notarás:

- No descarga nada
- Solo verifica la firma digital

Docker confirma que tu imagen local es idéntica a la remota.

---

# 🌍 ¿De dónde vienen las imágenes?

Por defecto Docker descarga imágenes desde:

👉 **Docker Hub**

Docker Hub es el repositorio público más grande de imágenes Docker.

Ahí encontrarás:

- Bases de datos
- Servidores web
- Lenguajes
- Frameworks
- Herramientas listas para usar

---

## ✅ Imágenes Oficiales

Siempre que sea posible usa imágenes marcadas como:

✅ Official Image

Porque:

- Siguen buenas prácticas
- Son verificadas
- No contienen código malicioso
- Son mantenidas oficialmente

---

# 📂 Imágenes vs Contenedores

En Docker Desktop notarás dos secciones:

## 📦 Images
Contiene las imágenes descargadas.

Ejemplo:

    hello-world

Ocupan espacio en disco.

---

## 🧱 Containers
Cada ejecución crea un nuevo contenedor.

Ejemplo:

- container_1
- container_2
- container_3

Cada vez que ejecutas `run`, Docker crea uno nuevo.

---

# ⚠️ Algo Importante

Cada ejecución genera:

✅ Un contenedor nuevo
❌ No reutiliza el anterior

Por eso empezarás a acumular contenedores detenidos.

Más adelante aprenderemos a:

- Listarlos
- Eliminarlos
- Administrarlos manualmente

---

* docker run → crea contenedor
* contenedor → imagen ejecutándose
* Docker Hub → repositorio de imágenes

Este curso incluye muchos ejercicios porque:

# 📦 ¿Qué tenemos ahora?

Si abres **Docker Desktop**, notarás:

- Muchos contenedores
- Estado: `Exited`
- Ninguno está corriendo
- Varias ejecuciones de `hello-world`

Cada vez que ejecutamos:

    docker container run hello-world

Docker:

1. Crea un contenedor nuevo
2. Ejecuta la imagen
3. Termina el proceso
4. Deja el contenedor detenido

Resultado 👉 acumulación de contenedores.

---

# 📋 Ver Contenedores desde Terminal

Primero exploremos los comandos disponibles:

    docker container --help

Aquí encontrarás todas las operaciones posibles sobre contenedores.

---

## Listar contenedores activos

    docker container ls

⚠️ Solo muestra contenedores en ejecución.

---

## Mostrar TODOS los contenedores

    docker container ls -a

Ahora verás:

- Contenedores activos
- Contenedores detenidos
- Historial completo

---

## Nota de Retrocompatibilidad

También existe:

    docker ps

Hace exactamente lo mismo.

Docker mantiene este comando por compatibilidad con versiones antiguas.

---

# 🗑️ Eliminar Contenedores

Existen varias formas.

---

## ✅ Eliminar un contenedor específico

Usando el ID o nombre:

    docker container rm ID_CONTENEDOR

Ejemplo:

    docker container rm a57

Docker permite usar solo los primeros caracteres del ID.

---

## ✅ Eliminar varios contenedores

    docker container rm id1 id2 id3

Ejemplo:

    docker container rm 606 a57

---

## ✅ Eliminar TODOS los contenedores detenidos

Forma más rápida:

    docker container prune

Docker mostrará una advertencia:

    WARNING! This will remove all stopped containers.

Confirma con:

    y

✅ Todos los contenedores detenidos serán eliminados.

---

## Verificar limpieza

Ejecuta:

    docker container ls -a

No deberías ver contenedores.

---

# 📦 Las Imágenes Siguen Existiendo

Aunque eliminamos contenedores, la imagen aún permanece localmente.

Puedes verla en Docker Desktop o con:

    docker image ls

---

# 🗑️ Eliminar Imágenes

Docker mantiene la misma lógica de comandos.

Si para contenedores usamos:

    docker container rm

Para imágenes usamos:

    docker image rm

---

## Eliminar imagen por nombre

    docker image rm hello-world

---

## También puedes usar el ID

    docker image rm IMAGE_ID

Docker eliminará:

- La etiqueta (tag)
- La imagen almacenada localmente

---

## Verificar eliminación

    docker image ls

La lista debería quedar vacía ✅

---

# 🧠 Patrón Mental de Docker (MUY IMPORTANTE)

Docker intenta mantener consistencia:

| Recurso | Listar | Eliminar |
|---|---|---|
| Contenedores | docker container ls | docker container rm |
| Imágenes | docker image ls | docker image rm |
| Volúmenes | docker volume ls | docker volume rm |
| Redes | docker network ls | docker network rm |

👉 Aprende el patrón una vez y sabrás usar todo Docker.

# 🌐 Ejecutando Contenedores Web y Publicando Puertos en Docker

En esta clase comenzaremos a ejecutar un **contenedor real con una aplicación web** y entenderemos uno de los conceptos más importantes de Docker:

> 🔥 **La publicación de puertos**

---

# ✅ Paso 1 — Verificar el Puerto Local

Abre tu navegador web y escribe:

localhost


### Resultado esperado

✅ No debería aparecer nada
✅ Error de conexión normal

Esto significa que **no tienes nada corriendo en el puerto 80**.

---

## ⚠️ Si ya tienes algo en localhost

Puede existir un conflicto porque:

- Otro servidor usa el puerto 80
- Apache / Nginx / XAMPP está activo
- Alguna app local está ejecutándose

No te preocupes ✅
Docker permite cambiar fácilmente los puertos.

---

# 📦 Imagen: `docker/getting-started`

Usaremos una imagen oficial diseñada para aprender Docker.

Esta imagen contiene:

- Una aplicación web
- Tutorial interactivo
- Ejemplo práctico oficial

Se encuentra en **Docker Hub**.

---

# ▶️ Paso 2 — Ejecutar el Contenedor

Ejecuta: `docker container run docker/getting-started`


---

## 🔎 ¿Qué sucede?

Docker:

1. Busca la imagen localmente
2. Si no existe → la descarga
3. Descarga múltiples **capas**
4. Crea un contenedor
5. Ejecuta la aplicación

Cada capa posee un **hash único**.

Esto permite que Docker reutilice capas ya descargadas y sea mucho más rápido.

---

## ❗ Problema

Si visitas nuevamente: localhost


👉 No verás nada.

¿Por qué?

Porque **el contenedor no tiene puertos publicados**.

---

## 🧠 Analogía

Es como tener:

🎳 Una bola de boliche sin agujeros.

Existe… pero no puedes usarla.

---

# ⛔ Detener el Contenedor

Presiona: CTRL + C


El contenedor se detendrá.

---

# 🚀 Paso 3 — Ejecutar en Segundo Plano

Ahora ejecutaremos el contenedor en modo *detached*: `docker container run -d docker/getting-started`


### `-d` significa

✅ Detached mode
✅ Corre en segundo plano
✅ Libera la terminal

Verifica: `docker container ls`


El contenedor estará corriendo.

---

## ❗ Aún no funciona localhost

Porque todavía **no publicamos ningún puerto**.

---

# 🌉 Paso 4 — Publicar Puertos

Ejecuta: `docker container run -dp 80:80 docker/getting-started`


---

## 📌 Explicación

* -d → segundo plano
* -p → publish (publicar puerto)
* 80:80 → HOST:CONTENEDOR

Significa:

`Puerto 80 de tu PC → Puerto 80 del contenedor`


---

## ✅ Ahora prueba

Abre:

http://localhost


🎉 Verás la aplicación funcionando.

Docker conectó tu computadora con el contenedor.

---

# 🔄 Cambiar el Puerto Local

Si el puerto 80 está ocupado:

```
docker container run -dp 8080:80 docker/getting-started
```

Ahora accede a:

http://localhost:8080


---

## 🧠 Concepto Clave

Un contenedor está **aislado**.

Solo puede comunicarse con el exterior si publicamos puertos.

```
PC (8080) → Contenedor (80)
```

Puedes ejecutar múltiples instancias:

- localhost:8080
- localhost:3000
- localhost:5000

Todas usando la misma imagen.

---

# ⏹️ Detener un Contenedor

Primero obtén el ID:
```
docker container ls
```
Luego:
```
docker container stop ID_CONTENEDOR
```

---

# ▶️ Iniciar Nuevamente

Sin recrearlo:

```
docker container start ID_CONTENEDOR
```

El contenedor vuelve a ejecutarse.

---

# 🗑️ Eliminar Contenedor

No puedes borrar un contenedor en ejecución.

Primero:

```
docker container stop ID
```

Luego:
```
docker container rm ID
```

---

## 🔥 Eliminación Forzada

También puedes hacer:
```
docker container rm -f ID
```

Docker lo detiene y elimina automáticamente.

# 🌱 Variables de Entorno en Docker

En esta clase comenzaremos a trabajar con un concepto **fundamental en Docker y en el desarrollo moderno**:

> ✅ **Variables de entorno (Environment Variables)**

Son usadas constantemente cuando trabajamos con contenedores reales, especialmente con:

- Bases de datos
- APIs
- Backend services
- Aplicaciones multi-contenedor

---

# 🧠 ¿Qué es una Variable de Entorno?

Una **variable de entorno** es simplemente:

> Una configuración que existe dentro de un entorno de ejecución.

En Docker, ese entorno normalmente es:

👉 **el contenedor**

---

## 📦 Ejemplo conceptual

Un contenedor puede necesitar información como:

- contraseña de base de datos
- usuario
- puerto
- modo producción/desarrollo
- claves API

En lugar de escribirlas dentro del código, se envían como variables de entorno.

---

# ⚙️ ¿Para qué sirven?

Permiten configurar un contenedor **sin modificar la imagen**.

Ejemplo real:

- misma imagen ✅
- diferente configuración ✅
- distintos entornos ✅

```
Desarrollo
Testing
Producción
```


Todo usando la misma imagen.

---

# 🧩 Cómo pasar Variables de Entorno

Cuando ejecutamos un contenedor usamos: `-e NOMBRE_VARIABLE=valor`

Ejemplo general: `docker container run -e MI_VARIABLE=valor imagen`


---

## 📌 Importante

Las variables disponibles dependen de:

1. La imagen que estés usando
2. La documentación oficial
3. O las variables que tú definas al crear tu propia imagen

El contenedor es como:

> Una pelota de boliche sin agujeros.

significa:

👉 detener + eliminar automáticamente.

# 🧠 Algo Muy Importante Sobre Docker

Notarás que varias imágenes PostgreSQL pueden compartir el mismo:
```
IMAGE ID
```

Esto significa:

> Son la misma imagen con diferentes **tags**.

Docker solo les asigna distintos nombres.

- ID corto

o usando el ID:

docker image rm IMAGE_ID

# 🧠 Concepto Nuevo — Docker Layers

Cuando eliminas una imagen verás mensajes como:
```
Deleted layer...
```

Esto ocurre porque:

👉 Las imágenes Docker están construidas por **capas (layers)**.

Cada instrucción del `Dockerfile` crea una capa independiente.

Beneficios:

- reutilización
- descargas más rápidas
- menor uso de espacio

Más adelante veremos esto en detalle.
